# AGENTS.md

## 语言

- 所有说明、分析、计划、总结、测试结果统一使用中文。
- 默认优先中文提问、中文回复。
- 代码标识符、库名、接口字段名、命令参数可保留英文。

## 启动流程

- 每次开始任务时，先检查并读取项目中的 `AGENTS.md`。
- 若该文件不存在，则先完成项目结构与关键配置的初步阅读，再在项目内创建符合当前项目的 `AGENTS.md`。
- 后续所有任务优先遵循该文件中的约定；如与本文件冲突，以项目内约定为准。
- 开始修改前，先阅读相关代码、配置、类型定义与调用链。
- 如任务涉及前后端联动、构建、测试或脚手架，先确认项目技术栈与现有规范，再动手修改。

## 工作原则

- 优先最小必要改动。
- 优先复用现有实现、现有组件、现有工具函数和项目约定。
- 不修改与当前任务无关的文件。
- 不进行无必要重构，不为“顺手优化”扩大改动范围。
- 修改前先理解上下文，避免基于猜测直接改代码。
- 遇到不确定信息，先核对项目现状，再继续。

## 终端与环境

- 默认使用 `pwsh` 执行 PowerShell 命令。
- 默认使用 UTF-8 编码。
- 注意中文文件名、中文路径、中文输出的兼容性。
- 处理文件时优先使用不会破坏编码的方式。
- 在 Windows 环境下避免依赖 bash 特有行为，除非项目明确支持。

## 依赖管理

- Python 项目严格使用 `uv`。
- 非必要不新增依赖；能用现有依赖实现的，优先不引入新包。
- 新增依赖前先评估是否存在项目内替代方案。

## Python

- 必须使用类型标注。
- 保持函数职责单一。
- FastAPI 路由保持轻量。
- 业务逻辑放在 Service 层或等价业务层。
- 请求和响应优先使用 Pydantic。
- 代码应优先保证可测试性、可读性和明确的输入输出边界。

## API 联动

- 接口变更必须同步检查：
  - 前端调用
  - 类型定义
  - 请求封装
  - 页面展示
  - 服务端校验与返回结构
- 不允许留下前后端不一致的状态。
- 若接口字段、状态码、分页或错误结构变化，必须同步更新相关调用方。
- 若存在 mock、示例数据、文档、测试用例，也要一并检查。

## AI / LLM

- Prompt、配置、解析逻辑分离。
- 不静默修改输出格式。
- 不硬编码以下内容：
  - API Key
  - Token
  - 模型路径
  - 私有配置
- 需要修改提示词时，优先保留可配置化能力。
- 涉及模型输出解析时，优先增加显式校验与失败兜底。

## 验证

完成修改后，尽可能执行与当前改动最相关的验证。

### Python

- 运行最相关的测试或类型检查。
- 如项目已有专用测试命令，优先使用项目现有命令。

### Docker

```bash
docker compose config
docker compose build
docker compose up -d
docker compose ps
docker compose logs error-analysis-service
```

### 通用要求

- 未验证不得声称已修复。
- 无法验证时必须说明原因。
- 若验证失败，先定位失败原因，再决定是否继续修复。

## 禁止事项

- 不伪造测试结果。
- 不硬编码敏感信息。
- 不新增占位实现来冒充完成。
- 不引入无必要的破坏性改动。
- 不随意升级无关代码。
- 不为完成任务而绕过已有规范。
- 不在缺乏依据时推断项目约定。

## 回复格式

每次回复尽量使用以下结构：

### 改动概述

说明本次做了什么、为什么这么改。

### 修改文件

列出实际改动的文件。

### 关键实现

说明关键逻辑、设计取舍或联动影响。

### 验证结果

说明执行了哪些验证、结果如何。

### 风险说明

说明仍可能存在的风险、限制或待确认项。
